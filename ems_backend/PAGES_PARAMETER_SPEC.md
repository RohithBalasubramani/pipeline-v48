# Page-by-Page Derived Parameter Spec

Captures the **derived columns** each frontend page needs, beyond the
parameters already seeded in [`seed_parameters.py`](lt_panels/management/commands/seed_parameters.py).

Each section lists columns that the simulator should produce so the page
can render fully without runtime derivation. Status: spec only — not yet
seeded as `Parameter` rows.

---

## Universal convention — Time-series filters

**Every time-series widget on every page accepts the same filter set.** This is
already implemented by the existing history WebSocket consumers
(`voltage-history`, `current-history`, `demand-profile`, planned
`thermal-history` / `loss-history` / `alarm-frequency`).

### Range filter (one of)

| Param | Examples |
|---|---|
| `range` | `today`, `yesterday`, `this_week`, `this_month`, `last_24h`, `last_7d`, `last_30d` |
| `start` + `end` | ISO 8601 timestamps — overrides `range` when both present |

### Sampling filter

| Param | Values |
|---|---|
| `sampling` | `minute`, `5min`, `15min`, `30min`, `hour`, `day` |

### Refresh

| Param | Default |
|---|---|
| `refresh` | `30` seconds — interval at which the server recomputes and pushes the buckets |

### Live re-config without reconnecting

History consumers also accept a JSON message at any time to switch the filter:
```json
{"range": "this_week", "sampling": "day"}
```
or
```json
{"start": "2026-04-01T00:00:00Z", "end": "2026-04-30T23:59:59Z", "sampling": "hour"}
```
The server immediately resends a `snapshot` with the new params.

**Pages should call out time-series filter needs only when they deviate from
this universal set** (e.g., a non-standard sampling like `15s` for some
high-frequency stream, or an additional categorical filter like a tab
toggle that changes which columns are returned).

### D. NEW WS consumer: `alarm-frequency`

For the **T-H7: Alarm Frequency** widget. Counts alarm events by type over a `range` window — but computed at query time from existing time-series columns. **No new model needed.**

Standard history query params: `range`, `start`/`end`.

Derivation rules per alarm type:

| Alarm type | Source column | Logic |
|---|---|---|
| `thermal_overload` | `kpi_kw_load_pct_of_rated` (common 126) | count windows where value > 100% |
| `buchholz` | `buchholz_relay_status` (Overview spec G) | count transitions into `alarm` / `trip` |
| `wti_trip` | `winding_temperature_c` (Overview spec A) | count crossings above `hotspot_critical_temp_c` |
| `oti_alarm` | `top_oil_temperature_c` (Overview spec A) | count crossings above `oil_high_temp_c` |
| `overcurrent` | `current_max` (common 126) | count crossings above `rated_current_a` |
| `earth_fault` | `earth_fault_active` ← **new column above** | count transitions to true |

Response shape:
