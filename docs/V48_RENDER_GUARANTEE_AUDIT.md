# V48 Render-Guarantee Audit — every way a card can fail

Generated from an 8-dimension + completeness-critic audit against the LIVE DBs (neuract :5433, cmd_catalog/cmd_equipment :5432) and both V48 + backend2 code. Goal: the new nameplate/envelope/gate layer must cover EVERY mode below so every card renders correctly OR honest-blank-with-reason — never mock, wrong, or crash.

**62 failure modes / 8 dimensions.** Severity: critical=15, high=24, medium=16, low=7


## DATA-SOURCE

_Queried the live neuract schema (target_version1, :5433) and the V48 fill/consumer/derivation code. Of 308 gic_* tables, 104 (34%) are COMPLETELY EMPTY (0 rows) — including UPS-04/05/06/08/10/11/12, all solar incomers, all 6 DG gic tables, all 4 PCC transformer _sch tables, and ~40 spares — so any card whose resolved meter lands on one renders blank. Data is FRESH (latest row ~12s old) but the window is only ~6 days (2026-06-25 → 2026-07-01), so any 30d/90d/YTD card has no history before 06-25. The schema is NOT uniform: _p1=72 cols, _tm (600kVA UPS in areas 17/27)=56 cols with a COMPLETELY DIFFERENT UPS-specific naming (output_active_power_total_kw, battery_backup_pct — zero metric-name overlap with _p1), _ng/_se/_jk/_sch=70 cols (missing harmonic_5th/7th_pct); feedback tables (pcc_panel_N_feedbacks) carry only breaker/relay/temp signals, no power/energy at all. Whole columns are structurally never populated: harmonic_5th_pct, harmonic_7th_pct, thd_voltage_r/y/b_pct, thd_compliance_v_avg/ieee519 are NULL in 100% (115/115) of latest _p1 rows; thd_current_*_pct NULL in 70/185. 9 meters have active_energy_import_kwh=0 with all energy in active_energy_export_kwh (reversed CT) — so todaysEnergyTotalKwh (registry.py:54, derives from import) returns a WRONG 0 kWh. 12 meters show negative latest power (UPS discharge convention + gic_30_* denormalized-float garbage -4.6e-44). Nameplate is unrecoverable from DB (ups_config absent, lt_config_value empty) — only name-parse (CL:600KVA) works. Topology dummy points PCC-Panel-1 at 8 feeders of which 4 are empty (silent ~50% under-count). And DG assets are DUPLICATED in the registry: id 2 (dg_1_mfm, 11464 rows) vs id 300 (gic_28_n1_dg_01_jk, 0 rows) — picking the wrong one blanks the card. The V48 services.py already guards missing columns (get_table_columns + _pad_missing None) and the WS source honest-degrades (None, why), but the worker/fill/kinds are TODO stubs and fetch_hourly_buckets (services.py:531) queries non-existent demand_* columns unguarded (currently dead code). The new layer must: pre-flight has_data/has_values per resolved meter, prefer the populated duplicate, detect import≈0∧export>0 to switch energy registers, clamp/reject denormalized negatives, exclude empty feeders from aggregates with a coverage note, honest-degrade structurally-null columns with a reason, and resolve _tm/feedback assets via their own schema mappers — never fabricate._


### [DS-01] 104 of 308 gic_* tables are completely empty (0 rows) — resolved meter yields blank card
- **sev/behavior:** critical / silent_empty
- **trigger:** A card resolves to a meter whose gic table has 0 rows (e.g. UPS-04/05/06, solar incomers, DG gic tables, spares).
- **example:** gic_02_n5_ups_04_cl_600kva_p1 (registry id 23, name 'GIC-02-N5-UPS-04 CL:600KVA') = 0 rows; also gic_02_n6_ups_05, gic_02_n7_ups_06, gic_06_n4_ups_08, gic_07_n5..n7_ups_10..12, all solar_incomer_1/2, gic_28_n1..n6_dg_01..06_jk.
- **evidence:** count_empty.sql over information_schema: TOTAL=308 EMPTY=104; per-table SELECT count(*) FROM neuract.gic_02_n5_ups_04_cl_600kva_p1 = 0
- **coverage:** honest-degrade — the layer must call layer1b/resolve/has_data.tables_with_data (has_data.py:54) BEFORE emitting the card's consumer; if the resolved meter's table has 0 rows, render the card frame with an honest 'no data logged for <asset>' reason instead of an empty widget. Where a populated sibling/duplicate exists (see DS-09) prefer it; otherwise self-heal by re-resolving to the parent panel aggregate.

### [DS-02] Data window is only ~6 days — 30d/90d/YTD cards have no history
- **sev/behavior:** high / silent_empty
- **trigger:** A card requests a trailing window (last_30d, this_month, YTD) that starts before 2026-06-25.
- **example:** gic_01_n3_ups_01_p1 earliest row = 2026-06-25 19:34 IST, latest = 2026-07-01 11:48 IST. A 'last 30 days energy' card asking start=2026-06-01 gets fetch_energy_delta start_row=None → window_energy_kwh returns None (energy.py:19-23).
- **evidence:** SELECT min/max(timestamp_utc::timestamptz) FROM neuract.gic_01_n3_ups_01_p1 → 2026-06-25 .. 2026-07-01 (6 days, 41449 rows)
- **coverage:** honest-degrade + clamp — the layer must read the meter's actual min(ts), clamp the requested window to available range, and annotate the card 'data available from 2026-06-25' (via the same reason channel as ems_backend_source why). NEVER extrapolate synthetic history. For cumulative-energy deltas, honest-degrade to None when start_row is missing (already correct in energy.window_energy_kwh) and surface the reason to the frame.

Where no mapping exists for a requested metric, _pad_missing None + honest reason. Do NOT feed a _tm table through the _p1 mapper (it currently returns all-None because _select_existing drops every requested column).

### [DS-05] 9 meters have active_energy_import_kwh=0 with all energy in export register (reversed CT) — today-energy derives to a WRONG 0 kWh
- **sev/behavior:** critical / wrong_value
- **trigger:** todaysEnergyTotalKwh / windowEnergyKwh derives from active_energy_import_kwh on a reverse-polarity meter.
- **example:** gic_01_n3_ups_01_p1 latest: active_energy_import_kwh=0, active_energy_export_kwh=311194. energy.todays_energy_total_kwh (energy.py:26) computes max(a1-a0,0) on import → 0 kWh, hiding the real 311 MWh throughput. Same on ups_02/03, ups_07 (export=1.49M), mldb, incomer_5/6.
- **evidence:** energy_check.sql: 9 tables with import=0∧export>0; registry.py:53-55 wires windowEnergyKwh/todaysEnergyTotalKwh to active_energy_import_kwh only
- **coverage:** deterministic-rule — the layer must detect import≈0 ∧ export>0 for a meter and use active_energy_export_kwh as the energy register (these are net-import feeders with reversed CT). Add a register-selection step ahead of energy.window_energy_kwh that picks whichever cumulative column is monotonically increasing / non-zero. Tag fidelity real_exact but note 'export register (reversed CT)'. This is a value-correctness fix, not a blank — the card currently shows a confidently wrong 0.

### [DS-06] 12 meters show NEGATIVE latest active power; gic_30_* carry denormalized-float garbage (-4.6e-44)

## ENVELOPE-ROUTING

But the identical 'HONEST-DEGRADE' pattern — try{ live }catch{ keep p.data } — is in EVERY fill module. Concrete: /home/rohith/desktop/BFI/backend/layer2/pipeline_v48/host/web/src/cmd/fill/feeder-real-time-monitoring.tsx:78-92 (card 36 Power&Energy) and :96-110 (card 38 Current Monitor): on a missing/unmappable frame the catch falls back to p?.data / p?.freshness, which ARE the byte-identical Storybook args (registry.tsx:10 calls them 'the exact_metadata seed'). grep across host/web/src/cmd/fill/*.tsx shows all 9 modules carry this seed-fallback. This is the single highest-volume wrong-value path.
- **renderCmd() is called OUTSIDE the per-card error Boundary — a synchronous throw in a fill fn white-screens the whole app despite the boundary** — frontend-render dimension said 'no app-root boundary' and 'per-card Boundary catches CMD_V2 throws'. But it MISSED that in /home/rohith/desktop/BFI/backend/layer2/pipeline_v48/host/web/src/components/CmdCard.tsx the Boundary (line 6-13) wraps only the already-computed `node`, while `const node = renderCmd(...)` runs at line 44 BEFORE/OUTSIDE the Boundary. React error boundaries do NOT catch errors thrown during the parent's own render (CmdCard's render), only during a child's. So if a fill fn throws synchronously while building the element (e.g. dereferencing frame.widgets.x.y on a partial frame before returning JSX, or a bad unwrap()), it propagates up past CmdCard, and main.tsx:7-9 has NO app-root boundary → total white-screen. The new layer must wrap the renderCmd CALL itself in try/catch (or move it inside the Boundary child) AND add an app-root boundary, so one bad card never blanks the page.
- **has_data gate treats zeros and all-null-THD rows as data-bearing → idle/spare feeders and PQ-null feeders resolve and render meaningless real values** — data-source flagged export-only (import=0) and 104 empty tables. But the has_data gate in /home/rohith/desktop/BFI/backend/layer2/pipeline_v48/layer1b/resolve/has_data.py:33-35 counts columns where value <> 'null'::jsonb — so a ZERO is 'data'. Confirmed: gic_01_n1_spare_p1 latest row = active_power_total_kw 0, import 0, export 0, power_factor_total 1, harmonic_5th_pct NULL. Separately, live-reproduced PCC-Panel-1 PQ frame: pq_priority feeder 11 shows i_thd_pct:null, v_thd_pct:null but is presented as a 'good' row; header_kpis.pq_exposure.status='good' with avg_i_thd_pct:null — a PASS verdict computed from missing data. The new layer must distinguish 'has non-null values' from 'has meaningful values' (nonzero power OR nonzero energy delta) and must NOT emit pass/good verdicts when the underlying metric column is null — render those tiles honest-blank with reason 'metric not reported by this meter'.
- **SCADA/feedback tables (aux_hsd_plc_feedbacks, *_sch) are resolvable assets with ZERO electrical columns — cross-schema mismatch beyond the _tm UPS case** — data-source flagged only the _tm UPS 56-col schema. If a user browses the picker and pins AUX-HSD-PLC (or any *_sch) for an energy-power/voltage page, the CMD V2 mapper finds none of active_power_total_kw/voltage_avg → every card honest-blank at best, or a mapper doing arithmetic on undefined crashes. Live-measured: prompt 'power quality for PCC panel 1' → elapsed_ms 58907 for a SINGLE endpoint (fan-out over 8 feeders). A voltage-current page binds 3 endpoints (voltage-current + voltage-history + current-history per endpoint_registry HISTORY_BY_DOMAIN) → up to 180s serial. The new layer must fetch endpoints in PARALLEL, impose a per-request wall-clock budget, and RETURN partial frames + a per-endpoint 'timed out' reason so late endpoints degrade to honest-blank instead of zeroing the whole page.
- **Frame-fetch failure reason (why) is logged server-side but STRIPPED from the response — the card gets neither a frame nor a reason** — envelope-routing said 'why logged server-side but never reaches the card', but understated the mechanism and scope. In host/server.py:88-127, _card_frames computes (frame, ok, why) per endpoint but on not-ok ONLY calls stage(...) So a card whose endpoint frame failed gets frames[endpoint]=undefined → its fill hits the seed-fallback (mode 1) with zero user-facing reason. The new layer must thread a per-endpoint status map {endpoint:{ok,why}} into the response and into each card so the FE can render 'no live data — <why>' instead of a mock.
- **capacity/nameplate in the live frame is a fabricated placeholder (observed capacity:60 for PCC-Panel-1) that flows into % loading KPIs** — registry-nameplate said 'V48 lt_mfm has zero rated_capacity_kva'. The new layer must NEVER present loading% against a placeholder capacity; if no per-asset nameplate exists it must blank the % tile with reason 'rated capacity unknown for this asset', and treat the frame's capacity field as untrusted.

### Verdict
The 8 dimensions are individually thorough but leave real, live-reproducible holes. The single biggest MISSED failure mode is systemic, not card-specific: EVERY one of the 9 frontend fill modules degrades a missing/slow/empty frame to the byte-identical Storybook SEED payload with NO disclosure (the audit caught this on only 1-2 named cards). Combined with (a) the host DROPPING non-ok frames and providing NO reason channel back to the card, (b) the per-card error Boundary sitting OUTSIDE the renderCmd() call so a synchronous fill throw white-screens the whole app (no app-root boundary), and (c) the has_data gate accepting zeros/null-THD as 'data', a card on an empty or partial asset renders confident MOCK or a fabricated PASS instead of an honest blank. I confirmed the triple-gap live: PCC-Panel-1 (id 317) resolves has_data=True, returns a 58.9s frame (near the 60s ceiling; a 3-endpoint page can hit ~180s serial and drop everything), reports a bogus capacity:60 nameplate, and presents an 8-feeder PQ panel where 4 feeders are empty and all THD metrics are null as 'good/watch'. The new layer MUST: (1) forbid seed-payload from reaching a live card (watermark or force-blank + reason), (2) thread a per-endpoint {ok,why} status into the response and each card, (3) wrap the renderCmd call itself + add an app-root boundary, (4) redefine has_data as 'meaningful (nonzero/non-null-for-the-routed-metric) data' and add a per-(asset,page) column-class feasibility gate, (5) fetch endpoints in parallel under a wall-clock budget returning partial frames, and (6) treat the frame's capacity field as untrusted, blanking loading% when no per-asset nameplate exists (nameplate_config has only 1 global default, no per-asset kVA)."
