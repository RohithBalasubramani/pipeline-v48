# EMIT CORRECTNESS DESIGN — AI-first per-slot binding scheme (Layer-2 emit)

Spec status: **DESIGN ONLY — build AFTER approval.** Nothing here is implemented.

Goal: make the Layer-2 emit AI (`layer2/emit/emit.py::emit` → `layer2/prompts/data_instructions.md`) bind the
**correct** `data_instructions.fields[]` for every one of the 9 scenarios, primarily through **prompt + context
structure**, not a downstream code guard. Fix order honored: prompt/context → DB → code-backstop (last resort, one
narrow rule). No per-card / per-metric hardcoding — every mechanism is generic and data-driven.

## 0. GROUND TRUTH (verified in code/DB before designing)

- **S2 root cause is `layer1b/basket/describe.py:17`**:
  `_DERIVED = re.compile(r"(_pct$|_deviation_|_spread$|_unbalance|^current_(min|max)$|_loss_)")`.
  `describe('current_max_spread') → ['Current Max Spread', 'derived', 'A']` and
  `describe('current_unbalance_pct') → ['Current Unbalance Pct', 'derived', '%']` — **both are real live columns**
  (verified present in `gic_01_n3_ups_01_p1` et al. in `neuract`/CONSUMER_SCHEMA), yet the name-regex tags them
  `kind="derived"`. `describe('neutral_phase_ratio') → [...,'raw','']` (raw, no unit).
- **The basket line** (`layer2/emit/user_message.py:26-28`, `_basket_lines`) renders
  `  <column> | <metric> | <kind> | <unit> | <Y|N has_data> | <rank>`. So `current_max_spread` shows up as
  `current_max_spread | ... | derived | A | Y | ...`. The AI reads the literal token `derived` as an instruction to
  wrap it in an `fn`, and (S2) invents `currentMaxSpread` — which is **not** one of the 48 catalog fns
  (`catalog()` verified: 48 fns, `currentMaxSpread` absent). The gate accepts any `derived` field that merely carries
  `fn` + `base_columns` (`layer2/gates.py:185-190`) — it never checks `fn ∈ catalog()` — so the leaf silently
  honest-blanks at fill time (`registry.run('currentMaxSpread', ...) → None`).
- **The slot catalog** (`layer2/emit/slot_catalog.py::build_slot_catalog` + `render_slot_catalog`) **deliberately
  surfaces NO suggested column** (docstring lines 178-184: "does NOT rank/suggest columns (that machinery mis-led the
  AI)"). It DOES surface each leaf's verbatim `label`/`unit`/`section` context. Re-adding a *restricted, honest*
  candidate is the pivot of Approach B.
- **The recovery-library block** is generated by `layer2/emit/emit.py::_recovery_library_block()` from
  `ems_exec.derivations.registry.catalog()`; each line today is `<fn> | base_columns=[...] | fidelity=<...>`. There is
  **no `quantity=` tag** yet — Approach C/D's quantity-match needs it.
- **The code seam that already runs pre-gate** is `layer2/resolve/column_override.py::apply`, invoked at
  `layer2/build.py:209` BEFORE `gate_data_instructions`. It already mutates `fields[].column`, already emits
  **non-gating telemetry** (`di['_normalized']`), already handles mislabelled-const reclass and the slot-quantity
  unit-cross block. It is the natural, minimal home for the ONE code-backstop — it does NOT currently handle
  out-of-catalog `fn → raw` coercion.
- **`registry.run(value_key, ctx)`** returns `None` for any `value_key ∉ LIBRARY` (registry.py:234-238) — an invented
  fn cannot fabricate; it silently blanks. That is why S2 is *silent* today.

## 1. ADVERSARIAL STRESS — scenario × approach coverage grid

Legend: **✓ closes** · **~ closes with residual risk** · **✗ leaves the failure open**.
A = decision-procedure + closed-vocab + self-verify · B = basket-enrichment + pre-resolved candidates ·
C = target_column-bridge + per-slot isolation + reasoning · D = few-shot negatives + quantity-match + reflection.

| Scenario | A | B | C | D | Notes on residual risk |
|---|---|---|---|---|---|
| **S1** RAW exact-name (`current_r`) | ✓ | ✓ | ✓ | ✓ | Works today; every approach reinforces. |
| **S2** RAW label≠col (`max_spread`↔`current_max_spread`) — PRIMARY | ~ | ✓ | ✓ | ~ | A alone still lets the `derived` **basket tag** nudge the invent (prose demotion can be ignored — RAW-BEATS-FN already was). B removes the nudge at the source (data-bearing flag) AND hands the exact column (candidate). C's target_column-bridge turns "AI named `current_max_spread`" into a forced raw bind. D's negative helps but relies on recall. **B+C decisively close it; A/D are reinforcers.** |
| **S3** DERIVED real-fn (`neutralCurrent`) | ✓ | ✓ | ✓ | ✓ | D2 fails (no `current_neutral` data-bearing col) → D3 fn. All fine. |
| **S4** DERIVED no-fn-arithmetic (max−min, no lib fn) | ✓ | ~ | ✓ | ✓ | A's closed-vocab forbids inventing; C's bridge finds no raw col + no fn → D4; D's reflection deletes any slip. B's candidate is empty (honest) → routes to D3/D4. **All converge to honest-blank; correct.** DB tier can add a real fn if genuinely wanted. |
| **S5** PHYSICAL WALL (waveform/harmonics/nameplate-no-base/run-state) | ✓ | ✓ | ✓ | ✓ | Prompt already handles; keep the 4-walls text as the D4 terminal. B's candidate is empty. |
| **S6** SIBLING CONTAGION (raw middle tile between 2 derived tiles) | ~ | ✓ | ✓ | ~ | A/D's per-slot instruction is prose (can be batched over). B gives the *middle* slot its OWN candidate + data-bearing flag; C's **per-slot isolation** re-runs the procedure per slot with only that slot's context. **B+C close it structurally**; A/D reinforce. |
| **S7** WRONG-QUANTITY closest-fn (`neutralToPhaseRatioPct` on unbalance slot) | ~ | ✗ | ~ | ✓ | B is silent (candidates are raw-only). A/C need a quantity to compare — without the `quantity=` tag the compare is name-similarity (the failure mode). **D's `quantity=` enrichment + quantity-equality is the decisive mechanism**; A/C consume that tag. |
| **S8** MIS-TAGGED %-slot ← raw kW (card 76 `Load %` ← `active_power_total_kw`) | ~ | ~ | ✓ | ~ | The existing `column_override._slot_quantity` guard only fires for suffix keys (`…Kw/…Kwh/…Pct`) and only blocks a **kW/kWh/kVA**-into-`Pct` cross — but it operates on the *bound column* after the fact (honest-blank, not a correct bind). C's **%-slot rule inside D2** (a %-quantity slot's only valid raw col is a %-quantity col) prevents the bad bind at authoring time and routes to a rating fn. **C is the primary; the code guard is the backstop.** |
| **S9** %-NEEDS-DENOMINATOR (`rated_kva` NULL) | ~ | ~ | ✓ | ~ | Same %-slot rule (C) → route to `kpiKwLoadPctOfRated` whose `base_columns` include `nameplate:rated_kva`; when that base is absent from the basket, D3's base-columns-present test fails → D4 honest-blank. **C + D3 base-check close it.** |

**Reading of the grid.** No single approach closes all 9. The **decisive** closers are:
S2 → **B (data-bearing flag + pre-resolved candidate) + C (target_column-bridge)**; S6 → **B + C (per-slot
isolation)**; S7 → **D (`quantity=` tag + quantity-equality)**; S8/S9 → **C (%-slot rule) + D3 base-check**. A supplies
the **ordered spine** (so RAW is forced *first*) and the **self-verify** net; D supplies the **negatives** that teach
the exact three failure shapes. They compose without conflict.

## 2. THE MERGED SCHEME (one coherent, AI-first, generic design)

One decision spine (A) enriched by structural context (B), bridged by the AI's own naming (C), and armored by
quantity-match + negatives + reflection (D). Mechanism ownership:

1. **Ordered per-slot decision procedure D1→D4** (A) — RAW is step 2, *before* any fn is even considered; reaching the
   fn step is forbidden once the raw step fires. Turns RAW-BEATS-FN from ignored prose into a gate.
2. **`target_column`-bridge rule** (C) — the AI reliably *names* the right column in `target_column` already (given).
   Rule: **if the `target_column` you named is itself a data-bearing basket column, the field is `kind:raw
   column:<that>` — you may NOT wrap it in an fn.** This directly kills S2 by exploiting the one thing the AI does
   right.
3. **Data-bearing decoupling in the basket** (B) — the basket line stops presenting the misleading `derived`
   name-regex tag as an instruction; a `has_data=Y` column is flagged `★ DATA-BEARING` and its regex tag is renamed
   `name_hint=` and explicitly demoted. Removes the root nudge at the source.
4. **Pre-resolved per-slot RAW candidate** (B) — the slot catalog surfaces, per scalar slot, the single best
   data-bearing basket column by token overlap (restricted to `has_data=Y`), or nothing (honest). D2 becomes a
   *match*, not a *recall*.
5. **`quantity=` tag on every library fn + quantity-equality clause** (D) — the recovery-library block gains a
   `quantity=<family>` token; D3 requires the fn's quantity to EQUAL the slot's quantity (kills S7).
6. **%-slot rule inside D2** (C) — a percent/ratio/efficiency/loss slot's only valid raw column is a percent-quantity
   column; a raw kW/kVA column is never a valid raw match (kills S8); if no %-quantity col, go to the rating fn; if
   its rating base is absent, D4 (kills S9).
7. **Per-slot isolation clause** (C) — decide each slot on its OWN context; never inherit a sibling's decision (kills
   S6).
8. **Closed-vocabulary fn enforcement + few-shot negatives** (A + D) — the only legal `fn` values are the verbatim
   strings in the RECOVERY LIBRARY list; three named negatives (`currentMaxSpread`, `neutralToPhaseRatioPct`-on-
   unbalance, `active_power_total_kw`-into-`Load%`) teach the exact three failure shapes.
9. **Mandatory self-verification pass** (A + D reflection) — before emitting JSON, re-audit every field against the
   two closed vocabularies (fn ∈ library, column ∈ schema) and the quantity/%-rules; rewrite/drop violations in-model.
10. **ONE code-backstop** (last resort, S2-only) — in the *existing* `column_override.apply` seam: an out-of-catalog
    `derived` fn whose `target_column` IS a data-bearing basket column is coerced to `kind:raw`. Subordinate,
    telemetry-noted, fires ~never if 1–9 work.

**Conflict resolution / de-dup.** B's data-bearing flag and B's candidate both feed D2 — kept both (flag removes the
nudge; candidate supplies the exact name; they are complementary, not redundant). C's target_column-bridge and B's
candidate overlap on S2 but attack different failure legs (bridge = the AI already named it; candidate = the AI is
handed it) — kept both as belt-and-suspenders on the PRIMARY bug. A's self-verify and D's reflection are the same pass
— **merged into one** "MANDATORY SELF-VERIFICATION PASS". The %-slot logic lives in exactly one place (D2 sub-rule);
the code guard in `column_override._slot_quantity` stays as the pre-existing backstop, not duplicated in prose beyond
the D2 rule.

### 3(a) — Prompt edits to `layer2/prompts/data_instructions.md` (paste-ready markdown)

**Placement.** Insert the whole block below at the top of the `derived →` rules region — specifically **immediately
after line 73** (the `- derived → ...` lead-in) and **before the current `· ★ RAW-BEATS-FN:` bullet (line 74)**. The
existing `· ★ RAW-BEATS-FN` / walls / worked-examples bullets stay (they become the reference detail the procedure

````markdown
## THE PER-SLOT DECISION PROCEDURE — walk D1→D4 FOR EVERY SLOT, IN ORDER, INDEPENDENTLY, STOP AT THE FIRST HIT
Decide EACH slot in the "FILLABLE DATA-LEAF SLOTS" block on its OWN context. NEVER carry a sibling slot's decision:
a slot flanked by two derived tiles is NOT automatically derived — re-walk D1–D4 for it from scratch.

**D1 — NAME THE QUANTITY (you are reliably good at this — do it explicitly).**
From the slot's own `label` / `unit` / `section` (shown per slot in the FILLABLE DATA-LEAF SLOTS block), name the ONE
real column that would measure this slot's EXACT quantity. Call it `target_column`. (For a `%`/ratio/efficiency/loss
slot, the target quantity is that percent/ratio — NOT the underlying kW/kVA.)

**D2 — IS `target_column` A REAL DATA-BEARING COLUMN? → kind:raw. STOP.**
Look for `target_column` in the DB SCHEMA block.
- ★ **target_column-bridge:** if the column you named in D1 appears in the DB SCHEMA and its line is flagged
  `★ DATA-BEARING` (has_data=Y), emit `{kind:"raw", column:"<target_column verbatim>", ...}` and **STOP** — do NOT
  consider any fn, **EVEN IF the column's `name_hint` says "derived"**. A column that carries data is RAW-BINDABLE
  regardless of its name pattern. `current_max_spread`, `current_unbalance_pct`, `neutral_phase_ratio` are REAL LOGGED
  COLUMNS — bind them `kind:raw` by name; NEVER wrap a data-bearing column in an fn.
- ★ **`name_hint=` is a NAME-PATTERN hint, NOT an order.** A column ending `_spread`/`_unbalance`/`_pct`/`_min`/`_max`
  gets `name_hint=derived` purely from its spelling — that does not mean "compute it": if it is `★ DATA-BEARING`, it is
  already logged, so D2 binds it raw.
- ★ **If the slot line shows `raw_candidate=<col>`, THAT is your D2 match** — copy that exact column into `kind:raw`
  and STOP. (The candidate is the pipeline's own token-match on the schema; confirm its quantity fits, then bind it.)
- **% / RATIO / EFFICIENCY / LOSS slots — extra D2 guard.** For a slot whose label/key carries `%` / `Pct` / `ratio` /
  `efficiency` / `loss`, the ONLY raw column that satisfies D2 is one whose OWN quantity is that same percent/ratio
  (e.g. `current_unbalance_pct` for a current-unbalance-% slot). A raw kW / kVA / kWh column is **NEVER** a valid D2
  match for a %-labelled slot (binding `active_power_total_kw` into a `Load %` slot renders "Load 550320 %"). If no
  percent-quantity column exists, DO NOT bind raw — go to D3.
- A recovery fn is a LAST resort reached ONLY when D2 fails. If D2 matched, emitting a `derived` field is a DEFECT.

**D3 — NO DATA-BEARING COLUMN: DOES A REAL LIBRARY FN COMPUTE THIS EXACT QUANTITY? → kind:derived.**
Reach D3 ONLY if D2 found no data-bearing column for the quantity. Then:
- **Closed vocabulary (VERBATIM copy).** The ONLY legal values of `fields[].fn` are the exact strings enumerated in the
  RECOVERY LIBRARY list below. To emit `kind:derived` you MUST be able to point at the fn's line in that list and copy
  the name **character-for-character**. An fn you compose yourself — `currentMaxSpread`, `currentSpread`, `maxSpread`,
  `currentMaxSpreadPct`, anything not on the list — **DOES NOT EXIST**; naming it silently blanks the leaf. If you
  cannot find the exact line, you are FORBIDDEN to emit derived: return to D2, else go to D4.
- **Quantity-equality.** The fn qualifies ONLY if BOTH hold: (a) its `quantity=` tag EQUALS this slot's quantity, and
  (b) ALL its `base_columns` are in the DB SCHEMA. A fn whose quantity differs is a WRONG-QUANTITY binding even if it
  sounds close and its columns are present — e.g. `neutralToPhaseRatioPct` (`quantity=neutral-to-phase-ratio`) is WRONG
  on a current-UNBALANCE slot. Do NOT reach for the nearest-sounding fn.
- Emit `{kind:"derived", fn:"<verbatim library fn>", base_columns:[real schema cols], target_column:"<frame column>",
  scope:"row", agg:"derived"}`.

**D4 — NEITHER: HONEST-DEGRADE (emit NO field for this slot).**
No data-bearing column (D2 failed) AND no library fn whose `quantity=` equals the slot's quantity with all
`base_columns` present (D3 failed) → emit NO field. This covers the FOUR PHYSICAL WALLS (waveform / per-order
harmonics / nameplate-when-base-absent / run-state) and any quantity this meter does not measure or compute. Never
invent an fn, never invent a formula, never bind a different-quantity column. Set `answerability="partial"` and say why
in `data_note`. **A %/efficiency/ratio slot NEVER degrades to a raw kW/kVA column under a % label** — if its
denominator (`rated_kva`) is absent from the DB SCHEMA, it honest-degrades here (D4); it does NOT show raw kW.

### THREE NEGATIVES — the exact shapes that are WRONG (do not produce these)
1. WRONG (S2): slot "Worst Spread" (unit A), `current_max_spread` is `★ DATA-BEARING` →
   `{"kind":"derived","fn":"currentMaxSpread",...}`. `currentMaxSpread` is NOT in the library → the leaf blanks.
   RIGHT: `{"kind":"raw","column":"current_max_spread","unit":"A","agg":"last","source":"live"}`.
2. WRONG (S7): current-unbalance-% slot → `{"kind":"derived","fn":"neutralToPhaseRatioPct",...}` — that fn's
   `quantity` is a neutral-to-phase ratio, not current unbalance. RIGHT: if `current_unbalance_pct` is DATA-BEARING,
   D2 binds it raw; else D4 honest-blank (no fn computes current-unbalance-%).
3. WRONG (S8): slot `Load %` → `{"kind":"raw","column":"active_power_total_kw","source":"live"}` renders
   "Load 550320 %". RIGHT: D3 `{"kind":"derived","fn":"kpiKwLoadPctOfRated","base_columns":["active_power_total_kw",
   "nameplate:rated_kva"],"target_column":"kpi_kw_load_pct_of_rated",...}` — and if `nameplate:rated_kva` is absent,
   D4 honest-blank (never raw kW under a % label).
````

### 3(b) — Context / basket / user_message changes (generic, data-driven, buildable)

**(B1) Data-bearing decoupling in the basket line — `layer2/emit/user_message.py::_basket_lines` (lines 24-34).**
Change the rendered line so the misleading `describe.py` `kind` is presented as `name_hint=` (a demoted pattern hint),
and any `has_data=Y` column is explicitly flagged raw-bindable. New line shape (order preserved for readability):

```
  <column> | <metric> | name_hint=<raw|derived|event> | <unit> | data=<Y|N> | rank
```

and append to any `data=Y` line the marker:
```
  ★ DATA-BEARING — real logged value; bind kind:raw by this exact name (name_hint is a NAME-PATTERN hint, not an order)
```
Driven entirely by the existing `c.get('has_data')` (already computed by `layer1b/basket/col_dict.py::window_nonnull`)
and `c.get('kind')` (from `describe.py`). Zero per-column logic. Add a one-line header note in the DB SCHEMA preamble
(`user_message.py:112-113`): "`name_hint` is a NAME-PATTERN hint from the column's spelling, NOT an instruction — a
`★ DATA-BEARING` column is raw-bindable no matter what the hint says."

**(B2) Pre-resolved per-slot RAW candidate — `layer2/emit/slot_catalog.py`.**
Re-add ONE honest candidate per **scalar** slot (kind=="scalar"; NOT arrays, NOT bucket_series, NOT time-axis). Build
it generically:
- Token set of the slot = tokens of `{ctx.label ∪ ctx.unit ∪ ctx.section ∪ element_key ∪ last-key-of-slot-path}`,
  lowercased, split on non-alphanumerics (reuse the existing `_tok` helper, slot_catalog.py:18).
- Candidate pool = basket columns with `has_data=Y` **only** (honest — never suggest a null column).
- Score = size of token-overlap between the slot token set and the column's `{column-name tokens ∪ metric tokens ∪
  unit token}`. Require the overlap to include a **quantity token** (a unit token like `a`/`v`/`kw`/`%`, or a family
  token like `current`/`voltage`/`power`/`unbalance`/`spread`) so a bare `current` slot does not grab an arbitrary
  `current_*` column — this is the guard against the "noisy candidate" weakness.
- Attach `entry["raw_candidate"] = <best column>` when a unique top-scorer with a quantity-token overlap exists; else
  `None` (honest — routes the slot to D3/D4, exactly as the stripped design intended).
- `render_slot_catalog` appends `· raw_candidate=<col> (★ DATA-BEARING — D2 match)` to a scalar slot line when present;
  nothing when `None`.
This is pure token overlap over the real schema + `has_data` + the `unit`/family tokens already in the vocab —
**no per-card map, no metric dictionary beyond what is already loaded** (`config.vocab`). A leaf whose quantity has no
logged column gets an empty candidate → honest-degrade.

**(B3) `quantity=` tag on the recovery-library block — `layer2/emit/emit.py::_recovery_library_block` (lines 24-44).**
Add a `quantity=<family>` token per fn line, derived data-drivenly from the fn's descriptor (its `columns` / value_key
family in `registry.catalog()`), so D3's quantity-equality is a **structural** comparison, not name-similarity. New
line shape:
```
<fn> | quantity=<family> | base_columns=[...] | fidelity=<...>[  ★ topology-pair only ...]
```
The `<family>` is derived generically from the fn's output column family — the cleanest source is a new
`"quantity"` key on each descriptor in `registry.catalog()` (see 3(c)); the emit block just echoes it. Applies
uniformly to all 48 fns; adding/removing an fn changes only the generated block.

### 3(c) — DB / catalog rows

1. **`registry.catalog()` gains a per-fn `quantity` field.** In `ems_exec/derivations/registry.py::catalog()`
   (lines 182-186) add `"quantity": v.get("quantity")` to each dict, and add a `"quantity": "<family>"` to each
   descriptor in `_COMPAT`/`_NAMEPLATE` (the `_d(...)` helper, lines 44-45, gains an optional 6th field). Families are
   the fn's real output family, e.g. `neutralCurrent → "neutral-current"`, `neutralToPhaseRatioPct →
   "neutral-to-phase-ratio"`, `pfAngleDeg → "pf-angle"`, `kpiKwLoadPctOfRated → "load-percent-of-rated"`,
   `loadFactorPct → "load-factor-percent"`. **This is authored once, from the registry's own descriptors — not a
   per-card table.** (If `quantity` is left null on a descriptor, D3 quantity-equality simply can't match it → that fn
   is only pickable when the slot's D1 quantity token matches its name; safe default.)

2. **(Optional, S4)** if a genuinely-wanted arithmetic quantity has no fn (e.g. a real current max−min spread that is
   NOT logged as a column on some meter class), add ONE real fn to `_COMPAT` (e.g. `currentSpreadA` computing
   `max(current_r,current_y,current_b) − min(...)` with `quantity="current-spread"`). **Only if a real slot provably
   blanks that a human would consider answerable** — otherwise S4's correct behavior is the honest blank. This is a DB/
   catalog addition, not a code guard, and it makes the fn part of the closed vocabulary automatically.

3. No new `app_config` rows are required for the mechanism; the candidate-quantity token families reuse
   `config.vocab`/`config.metrics` already loaded. (If tuning is wanted later, the candidate's required-quantity-token
   set can be lifted to a `app_config emit.candidate_quantity_tokens` row — not needed for v1.)

### 3(d) — The ONE code-backstop (unavoidable, minimal, S2-only)

**Where:** `layer2/resolve/column_override.py::apply` (already invoked at `build.py:209`, already telemetry-noted,
already pre-gate). Add ONE branch, subordinate to everything above:

> For a field with `kind=="derived"` whose `fn` is **NOT** in `ems_exec.derivations.registry.catalog()` (set
> membership) **AND** whose `target_column` is a **real `has_data=Y` basket column**: rewrite the field to
> `{kind:"raw", column:<target_column>, source:"live"}`, drop `fn`/`base_columns`, and append a `_normalized` note
> `slot <s>: out-of-catalog fn <fn> coerced to raw on data-bearing target_column <col>`. Otherwise (fn out-of-catalog
> AND target not data-bearing) leave the field for D4/the gate to honest-blank (existing behavior).

**Why this is unavoidable (proof that prompt+context cannot *fully* close S2):** the self-verification pass (9) is
in-context reasoning, not a hard stop — a deterministic model that hallucinates "`currentMaxSpread` is in the list"
during its own audit passes its own check. Every prompt/context tier (B1 flag, B2 candidate, C bridge, A order, D
negative, self-verify) reduces the probability but none is a *guarantee*; only a set-membership check against
`catalog()` is. Because S2 is the PRIMARY, high-frequency bug and its failure is *silent* (blank leaf, `conforms=True`
today), a deterministic backstop is warranted. It is **generic** (set membership `fn ∈ catalog()` + `col ∈ basket &
has_data`, no card/metric/fn hardcoding), **subordinate** (fires only after all AI tiers), and **telemetry-visible**
(so sweeps count how often the AI still misses and the prompt can be tuned). It is the ONLY code-backstop; S3–S9 need
no code and are closed by prompt+context.

### 3(e) — FINAL scenario → handling matrix (all 9 covered)

| # | Scenario | Primary tier(s) | Mechanism that closes it | Residual risk |

1. Every `kind:"derived"` field's `fn ∈ set(e['fn'] for e in catalog())` — **no out-of-catalog fn ever ships**.
2. Every `kind:"raw"`/`"bucketed"` field's `column` ∈ the card's basket with `has_data=Y`.
3. No `%`/`Pct`/ratio/efficiency/loss slot is bound to a kW/kVA/kWh raw column.
4. `conforms` is not falsely `True` on a blanked leaf — a D4 slot carries an `_emit_gaps`/`data_note` reason.
