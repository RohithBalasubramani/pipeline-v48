════ HARD RULES R1–R14 — READ FIRST. Each is deterministically GATE-ENFORCED: a violating field/leaf is blanked, reverted or rejected and never ships. Later sections defer to these; none is restated. ════

R1. SLOT VERBATIM (the #1 correctness rule) — every data field's `slot` MUST be COPIED VERBATIM from the "FILLABLE DATA-LEAF SLOTS" block (the exact dotted/indexed leaf path, e.g. `health.data.phases[0].value`, `flow.vm.kpis.lossKwh`). An invented token (tile_r, kpi_actual, series_r, sourceInputKw, instantaneous_power) resolves to NOTHING — the field, even with a perfect column, is DISCARDED and the leaf stays empty. NEVER emit a slot not on that list. ONE field per leaf; a per-phase/per-series leaf gets one field per element path ([0].value, [1].value, …), each bound to THAT phase's column (R→*_r, Y→*_y, B→*_b, N→*_neutral).

R2. HONEST-BLANK = OMIT THE FIELD *AND* DECLARE IT (the ONE honest-blank convention; the #1 rule of this seam) — when NO basket column (and no legal fn per R14) measures a slot's OWN quantity — including the R6 walls and off-domain quantities this meter class does not measure (battery SOC / battery temperature / fuel level on an electrical meter) — emit NO field for that leaf AT ALL: the skeleton preserves the key, the leaf renders an honest blank with a per-leaf reason, data_note says why, answerability="partial". You MUST NOT instead: (a) bind a proxy fn/column of a DIFFERENT quantity (loadFactorPct into a "transfer readiness score"; activeEnergyTodayKwh into a "transfers count / days-since" slot); (b) ship a CONSTANT stand-in; (c) REUSE a value from another slot; (d) emit a raw/bucketed field with column=null (the gate defect 'missing a resolved column', not a blank — kind="time" is the ONLY legitimately column-less kind); (e) honest-blank via a mislabelled const. The CONVERSE is the #1 defect: do NOT omit a leaf when the schema HAS a matching column — that blanks a card that HAS data.
    R2-DECLARE — OMISSION ALONE IS NOT ENOUGH; you MUST ALSO ANNOUNCE the blank. Add a TOP-LEVEL `data_instructions._honest_blanked` array (beside fields[]) with ONE entry PER intentionally-blanked leaf — either the string `"<slot verbatim>: <plain reason>"` or the object `{"slot":"<slot verbatim>","reason":"<plain reason>"}` (both accepted; slot COPIED VERBATIM per R1, including any `[i]`/`[*]`). This is AUTHORITATIVE downstream: the executor reads `_honest_blanked` to know a leaf was blanked ON PURPOSE, so it will NOT mechanically rescue-fill it with a wrong-role/wrong-quantity value. A leaf you merely OMIT WITHOUT a `_honest_blanked` entry reads as an oversight, not intent — EVIDENCE: an `Output Voltage` slot left at its default `'—'` with no declaration, so the executor could not tell the blank was deliberate. `_honest_blanked` is MERGED with (never replaces) the gate's own wall records, so always declare your own. If you honest-blank ANY leaf you MUST list it here; `[]`/omit ONLY when you blank nothing.

R3. NEVER FABRICATE + COUNTABLE COVERAGE — never fabricate a column, number, fn, metadata key or frame field, not even to force answerability "full" (a substitute is a REAL column; a true gap is reported as "none"). Every listed slot is either (a) bound by EXACTLY ONE field or (b) deliberately omitted per R2 WITH the reason in data_note — an unexplained uncovered slot is recorded as a per-leaf gap; never silently ignore a slot, never pad coverage. NEVER A SILENT ZERO-SKELETON: binding NOTHING (fields [] and no roster) with no honest reason renders the skeleton's 0/[] placeholders as real values (a false "0 issues" story) — whenever you bind nothing, declare answerability "none"/"partial" AND a data_note naming exactly WHY; your authored reason beats the generated one.

R4. QUANTITY WALL — every DB SCHEMA line shows the column's `qty=`, every slot line its `expected_qty=`: a column of ONE physical quantity NEVER fills a slot of ANOTHER. power/energy is NOT a temperature (hotspotC/oilC/windingC), an aging factor (faa), a readiness/score, a count (tapCount/transfers), a duration (runHours/backup minutes), or a lifetime (lifeYears); a deviation/spread column is NEVER crest-factor/flicker (flickerPst); thd_current_* is NEVER a voltage-harmonic (h5/h7) or voltage-THD value. SHARING A UNIT DOES NOT MAKE TWO QUANTITIES THE SAME: a voltage-THD leaf (%) never takes thd_current_* (also %); SOC % never takes load %; a `years`/`°C` leaf never takes an energy column/fn (how "10,156 years" happens). Match the slot's LABEL SEMANTIC (voltage vs current, energy vs years), not the unit sign; a slot marked `(weak)` is dimension-only — its label decides. No column of the slot's quantity → R2; the gate blanks every cross-quantity bind ("<quantity> not measured by this meter"), and such a leaf is never "full".

R5. SEMANTIC-FAMILY WALL (the same-DIMENSION pun) — a slot whose label/metric/key names efficiency / SFC / specific consumption / fuel binds ONLY a source of that SAME semantic family — sharing '%' does not make a load factor an efficiency (loadFactorPct into an 'Efficiency' KPI rendered "Efficiency 5.3 %"). No same-family source (a DG with no fuel-flow measurement cannot measure efficiency/SFC) → R2, naming the missing family in data_note.

R6. THE FIVE PHYSICAL WALLS — quantities the meter does NOT log; NO fn fabricates them from unrelated columns; always R2, never a derived field: (1) waveform shape (crest_factor / flicker_pst); (2) per-order harmonics (harmonic_5th/7th_pct, k_factor); (3) externally-assigned nameplate/commercial figures (kpi_kw_load_pct_of_rated, rated_kva, subsidy/contract) when the base_columns are absent OR this asset's nameplate denominator is empty; (4) device run-state (breaker_state, *_trend_status); (5) TRANSFER/READINESS/SYNC/PERMISSIVE/BYPASS SCORES and TRANSFER COUNTS / DAYS-SINCE / LIFETIME — no fn turns a load factor or an energy counter into a readiness score or a transfer count.

R7. THE CANONICAL SAME-QUANTITY-FAMILY PROXY RULE (the ONE exception to locked display metadata — stated once, here): a substitute/proxy is legal ONLY within the slot's physical-quantity family — the bound column's `qty=` EQUALS the slot's `expected_qty=`: a coarser aggregate or adjacent form of the SAME measure ('per-phase kW' absent → total kW ✓; today's kWh absent → this-week kWh ✓; import register dead on a reversed-CT feeder → the EXPORT register delta ✓). SAME family: bind it (kind=raw/derived on a REAL column — NEVER a valueless const) + declare it in `data_note` + MORPH every caption/label/window leaf that DESCRIBES that slot to the proxy's TRUE meaning (e.g. `sourceInputCaption` 'SOURCE INPUT'→'MEMBER IMPORT', 'today'→'this week'), each listed in `_morphed` (siblings sharing the leaf → morph the per-slot caption instead). DIFFERENT family: FORBIDDEN, morphed or not — no data_note legitimizes it → R2. A declared proxy missing its display morph is gate-DROPPED (the leaf blanks rather than show a wrong reading under the default caption); an undeclared proxy is a UNIT-MISMATCH defect. PLAN PROXIES FIRST: exact_metadata is emitted BEFORE data_instructions — decide every proxy before authoring the metadata.

R8. NO BINDING REUSE ACROSS DISTINCT SCALARS — NEVER bind the SAME fn or the SAME column into ≥2 DISTINCT scalar VALUE slots naming DIFFERENT quantities (three "score" cells ups_input/bypass/sync_permissive_score; three "activity" cells days-since / count-30d / lifetime): copying one measurement into all three FABRICATES three distinct readings (they all render 96.3). Bind each scalar ONLY to the source measuring ITS quantity; the rest → R2, never padded. (LEGITIMATE: a SERIES and its own legend/kpi/axis-min/max summary share the series' column — one quantity across related slots; the defect is ≥2 DISTINCT-quantity SCALAR cells co-bound with NO co-bound series.)
    R8-ROLE — OWN-POINT / SAME QUANTITY, DIFFERENT SOURCE ROLE is the same defect: a SINGLE METER MEASURES ONLY ITS OWN MEASUREMENT POINT. A slot whose LABEL names a DEDICATED SENSING RAIL — bypass / input / mains / utility / grid / incoming / line-side / output / battery / rectifier — needs a column measured AT THAT rail; a meter that does not sense that rail has NO column for it. This gic_* meter senses only its OWN measured terminal — voltage_avg / current_avg / frequency_hz ARE that one reading (an INPUT reading on a UPS input meter; an OUTPUT reading on an output meter). NEVER bind the meter's own voltage/current/frequency into a leaf naming a DIFFERENT rail than the one it measures: `Average Bypass Voltage` ← the meter's voltage_avg is FABRICATION — it shows the meter's own reading under a bypass caption the meter cannot see (a same-quantity role smear the quantity wall R4 does NOT catch — voltage↔voltage is compatible; identical for freq↔freq, current↔current). Bind a rail-qualified leaf ONLY to a column whose OWN name carries that SAME rail (bind the plain reading ONLY to the leaf naming the rail this meter actually senses — input* on an input meter, output* on an output meter); every bypass* / mains* / utility* / grid* / incoming* / line-side* / battery* / rectifier* slot with no dedicated column for THAT rail → R2 honest-blank AND an explicit `_honest_blanked` entry per R2 (reason: "this meter has no <rail> sensor — it measures only its own <own-rail> point"). A time-axis label leaf (points[*].label / .slot) is ALWAYS kind=time (no column) — a measured column there is R2/gate-blanked (the c59 negative-kW-as-time-labels defect). The EQUIPMENT facts line's bay_role (incoming/outgoing/coupler/spare) names THIS meter's OWN measurement point: an incoming-role meter legitimately fills an incomer/supply-labelled slot ON ITS OWN ASSET — but a bay_role NEVER licenses borrowing ANOTHER meter's rail for a dedicated-rail slot this meter does not measure.

R9. SIBLING-SLICE + PHASE-AS-GROUP PROXY — sibling slices carrying DIFFERENT entity labels (breakdown[0..2].value labelled "UPS"/"BPDB"/"HHF"; every sections[i].totalKw; a sankey's nodes[i].value and links[i].value; sources/consumers[i].meters[j]) each claim a DIFFERENT sub-group's number: ONE panel-total column bound into all renders the same value N times and the groups sum to N× the panel (the 'every ribbon = 550,320 kWh' defect). The EVASION is equally forbidden: per-PHASE columns (*_r/*_y/*_b/*_neutral) bound one-each into GROUP-labelled slices — a PHASE is NOT a feeder group / member set / section; a per-phase column fills ONLY a phase-labelled leaf. No per-group column → bind the TOTAL slot only, slices unbound (R2) + answerability="partial" + data_note — on a PANEL asset (see the PANEL MEMBERS block) the panel-aggregate renderer fills each per-member leaf from that member meter's OWN row, so the blank still renders real per-member truth. (DISTINCT: the same column across TIME samples of one series is legitimate.)

R10. CONST-SOURCE — a NUMERIC const in a data slot ships ONLY when the field itself cites a REAL DB source: EITHER (a) a NAMEPLATE RATING — `metric` is exactly one of contracted_capacity_kw / rated_kw / rated_kva / rated_current_a; the executor substitutes this asset's REAL rating or honest-blanks the leaf; the baked `value` is only a shape placeholder (a PER-ASSET rating/contract is NEVER a literal — the default's number is a Storybook seed; a home-made spelling like I_RATED / I_MAX / deratedKva does NOT resolve and is blanked); OR (b) an app_config `consts.<name>` row — `metric` names the row (stress_border_pct, hotspot_warn_c) and `value` IS that row's value. Any other invented numeric literal (a guessed rated current 131, a derated 1000 kVA, made-up thresholds/axis ticks) is gate-blanked; no real source → R2. A UNIVERSAL standard (the IEEE_519 THD limit) is a true literal. An RTM STATUS-BAND boundary quoted VERBATIM from the RTM STATUS BANDS facts line (a stored cmd_catalog consts.rtm_<panel_type>_<metric>_<band> row) is likewise a REAL DB source — cite it as-is; a band/threshold value NOT on that facts line stays a guessed threshold (gate-blanked).
    R10-SEED — NEVER SHIP A DEFAULT (seed-strip): every NUMERIC LITERAL sitting in a DATA slot of the card's STATIC default — a legend value, a chart marker, a series point, a KPI number, a band/scale value — is a STORYBOOK SEED, NOT this asset's reading. It MUST be resolved to one of two truths and NEVER left at its seed value: EITHER (1) FILLED from a real basket column / library derivation (kind=raw/bucketed/derived on this asset's own-quantity column — the seed's slot gets a field), OR (2) explicitly HONEST-BLANKED per R2 (omit the field + a `_honest_blanked` entry). A seed literal that survives into the emitted payload renders a FIXTURE number as a live reading (fabrication). NEGATIVE EXAMPLE: a legend's `legendValue` seeded `[52, 71, 85, 43]` (or a marker `value: 96.3`) must NOT ship as-is — bind each legend/marker leaf to its real column (kind=raw), or honest-blank it; leaving the `[52,71,85,43]` seed is a defect (a MEASURED legend/marker is never a const per the const-kind rule below). Applies to EVERY seed numeric in a data leaf, not just the ones the story names.

R11. WINDOW/LABEL COHERENCE — the fetch range/window you declare, the `window.lookback` you emit, and EVERY period-declaring metadata leaf (`periodLabel`, `range`, a "Monthly"/"Weekly"/"Today" caption — NOT a picker's rangeOptions list) MUST all name the SAME window, the one the consumer actually fills from. A 'Monthly' label over a 24h fill mislabels every number under it (fabrication by caption). Either declare the range the label promises, or MORPH the label leaf to the fetched window's truth (in `_morphed`) — else the gate (policy `gates.window_label_policy`) morphs/blanks it and records the incoherence. Your declared range also DRIVES the backfilled fill window — declare the range the story needs, once, and keep every label consistent with it.

R12. BYTE-IDENTICAL METADATA + `_morphed` — author every exact_metadata leaf BYTE-IDENTICAL to the card's STATIC CONFIG default unless the prompt + 1a story justify a morph; `_morphed` (REQUIRED, inside exact_metadata) lists EVERY changed key path (`[]` when none). The producer applies ONLY declared morphs — an UNDECLARED morph is REVERTED to the default (silent no-op, recorded). HARDCODED/STATIC items are LOCKED, never morphed: legends (statusLegend), colour tokens/palettes/statusColors/selectionColors, axis labels (metricAxisLabels), units, descriptors, vocab, playback glyphs, shade/legend rows (e.g. the heatmap FOOTER's time-axis + shade legend), badge labels — byte-identical ALWAYS, NEVER in `_morphed`. MORPH ONLY THE STORY-DRIVEN FEW — lead with the prompt's metric (tab order), re-title, tighten/loosen a threshold the prompt flags, re-roster to the asked entity — and every morph must quote a word from the page story / this card's story angle; WHEN IN DOUBT, DO NOT MORPH (a morph on a locked item is rejected by the byte-identity gate). REQUIRED morphs, not optional: R7 proxy display leaves, R11 period labels, seed-roster identifiers (PART 2).

R13. DATA-RESIDENCE — $ctx IS NOT A WALL BYPASS. `is_group_card` sets the residence; there is NO emit_mode branch — the SAME output shape carries every card, `$ctx` only selects where the DATA lives. GROUP card → `$ctx` = the page's shared_context.$id; emit a LEAN ATOM: no baked data; EVERY data field source='$ctx' (reads the shared buffer by metric key — the column need NOT be a basket column; source='live' there triggers the basket check and flags the key hallucinated); each field carries `selection_role`; the atom STILL carries its OWN FULL exact_metadata AND its own fields[] (the buffer-projection list); interaction seeds you reference must exist in shared_context.interaction; FUNCTIONS NEVER travel. A group atom NEVER binds or substitutes a basket column (not even to "best-effort"); its answerability is "full" unless the buffer itself lacks the metric. STANDALONE card → `$ctx` null; EVERY raw/event field source='live' (or test-db) bound to the resolved asset/table and a REAL basket column; source='$ctx' there is gate-REJECTED. Walls R4/R5/R6/R8/R10 apply to $ctx fields IDENTICALLY (a $ctx power key in a temperature/years/readiness slot, one key smeared across distinct-quantity cells, a sourceless $ctx const — blanked like a live bind); only a structural $ctx atom with no quantity on either side (a time atom, a group-context projection) is exempt by construction.

R14. RAW-BEATS-FN; LIBRARY-ONLY FNS; EMPTY DENOMINATOR — check FIRST, before any fn: if a COLUMN BASKET line measures the slot's quantity AND shows data=Y (the ★ REAL LOGGED marker), bind it kind=raw by its EXACT name and emit NO fn — EVEN IF its name_hint says "derived" (name_hint is a spelling guess, not an order: current_max_spread, current_unbalance_pct, current_min/current_max/current_avg are LOGGED columns — READ them raw). A SUMMARY STAT or AXIS-bound slot (a `stats[i].value`, maxY/minY, expectedMax/Min) is ALMOST ALWAYS one of these raw columns, NOT an fn: read the stat's `expected_qty=`/unit and LABEL and bind the same-quantity data=Y raw column it names — "Peak Current"→current_max (kind=raw), "Average Current"→current_avg, a min/minY→current_min, a spread stat→current_max_spread. Reaching for a name-lure fn here is the #1 stat defect and a DATA LOSS: worstPeakKw is quantity=peak-active-power-kw (WRONG on a "Peak Current" A-unit stat — power, not current), loadFactorPct is quantity=load-factor (WRONG on "Average Current"); the wall blanks such a bind even though current_max/current_avg are right there data=Y. THE target_column-bridge IS DECISIVE: the instant the frame column you would name as `target_column` is itself a data=Y basket column, STOP — DELETE the fn and re-emit the field as kind=raw with `column`=that exact column (never wrap a real logged column in an fn; {kind:derived,fn:worstPeakKw,target_column:current_max} still blanks on the fn's quantity, {kind:raw,column:current_max} fills). A recovery fn is a LAST resort, legal ONLY when BOTH hold: (a) its `quantity=` tag EQUALS the slot's own quantity (a differing quantity= is a WRONG-QUANTITY bind even if the name sounds close — neutralToPhaseRatioPct is quantity=neutral-to-phase-ratio-percent, WRONG on a current-unbalance slot), AND (b) ALL its base_columns are in the COLUMN BASKET. The ONLY legal `fn` values are the value_keys enumerated VERBATIM in the RECOVERY LIBRARY — a composed fn (currentMaxSpread, maxSpread, currentSpreadPct, …) DOES NOT EXIST and silently blanks the leaf ('derivation unbound'). A fn with a `nameplate:<rating>` base is bindable ONLY when this asset's rating is ACTUALLY populated — an empty rating is NEVER a denominator (→ R2). No fn satisfying (a)+(b) → R2; never the closest-sounding fn.

════ PART 1 — KEEP / SWAP ════
Decide whether the card in this slot tells its assigned analytical story, or whether one of this slot's SWAP CANDIDATES (same footprint, ±15% in width AND height) tells it clearly better. SCOPE: judge RELEVANCE-TO-THE-STORY + COHERENCE only — you do NOT judge whether the data exists or the query will succeed (data_instructions + the helper handle fillability). Be CONSERVATIVE: KEEP is the default; 0 swaps is a correct, healthy answer. A swap is honored ONLY if ALL THREE hold:
1. STORY-ANGLE RELEVANCE — the candidate serves THIS slot's assigned analytical_story MORE DIRECTLY than the current card, and you can quote the specific word/phrase from the story angle that proves it. A tie or a vague 'better fit' is a KEEP.
2. LISTED — the candidate is one of THIS slot's printed SWAP CANDIDATES (the only size-equivalent, render_real options). A card listed elsewhere or not at all is NOT a valid target.
3. NO DUPLICATE — the target must be NEW to the page: never a card already on the page, already in 1a's TEMPLATE CARD SET (template_card_ids), or already chosen as another slot's target. 1a's chosen cards are sacred.
INTERDEPENDENCY: a card printing link_type master-selector / shared-selection / cross-highlight is COUPLED — swappable ONLY as an all-or-nothing COMBO naming every linked partner's matching swap in `cascade`; if any partner has no valid listed candidate, KEEP the whole set.
Set action='swap' ONLY when all three hold AND confidence >= 0.9 AND `criterion` NAMES the concrete story-angle word the candidate serves better (vague criteria — 'better', 'more relevant', 'nicer', 'good fit' — score below 0.9 and become a KEEP).

== data_instructions (the parseable DATA-fill RECIPE) ==
A RESOLVED recipe — one resolved field per data slot, plus the fill envelope. Bind every field to a REAL COLUMN BASKET column by its `metric` (never every tile to one column — R8/R9). Envelope:
- payload_shape — which DATA tier to fill. orientation — time | entity | snapshot. entity_dim / selection_dim — what each row/series IS and what selection drives it. selection_role — the card's interdependency role (both|produces|consumes|emits|none), provisional.
- binding — { asset_id, table, ts_col, panel_id, nameplate_scope } (from 1b); null for a pure-$ctx group atom or a pure-const card. NO top-level `source` — fill source is PER-FIELD `fields[].source`.
- window — { lookback, sampling, time_mode } from the card controls (re-slice = re-bind only); coherent per R11.
- fetch — ★ the fetch spec that DRIVES the live backend: the helper opens `ws/mfm/<asset mfm_id>/<endpoint>/`; the consumer returns the card's DATA (incl. history time-series); the frontend feeds that frame to the CMD V2 card's OWN mapper. Knobs: { endpoint, window_seconds, interval_seconds, sample_count, range, start, end, sampling, metrics, selection }
  - endpoint — pick EXACTLY ONE from this card's `fetch ENDPOINT` hint (the authority), by ANALYTICAL INTENT, not keyword overlap: REAL-TIME card (live rolling values / scrubber / now-snapshot) → the LIVE screen + LIVE knobs; TREND/PROFILE/HISTORY card (values must CHANGE WHEN THE USER PICKS A DATE) → a DATE-CAPABLE history variant + DATE-WINDOW knobs (this makes the card date-navigable; if the hint lists NO history variant, the live screen serves history via range/sampling, e.g. `power-quality-summary`). Prefer the card's NATURAL endpoints line; a RELATED screen is fine if THIS card's story needs it (a PQ card plotting a current trend → current-history); an UNRELATED page's screen (real-time-monitoring on a power-quality card) is wrong.
      · THE CLOSED SET — ONLY these endpoints EXIST (emit EXACTLY one): {{LIVE_ENDPOINTS}}.
      · ★ RETIRED — DO NOT EXIST, NEVER emit: {{RETIRED_ENDPOINTS}}. ALL Harmonics/PQ data is the SINGLE live socket `power-quality-summary` (no PQ history variant). Anything outside the closed set lands on the catch-all and returns no data.
  - LIVE knobs: window_seconds (rolling reach, e.g. 30), interval_seconds (cadence, e.g. 2), sample_count (depth, e.g. 12); set range/start/end/sampling = null.
  - DATE-WINDOW knobs (THE DATE NAVIGATION SEAM): range = today | yesterday | last-7-days | this-month | custom-range (the DEFAULT window; the frontend OVERRIDES it on date change); start/end = ISO or bare YYYY-MM-DD, ONLY when range=custom-range; sampling = hourly | 2hour | shift | day | week (bucket width suits the range: today→hourly/2hour/shift; week→day; month→week); set window_seconds/interval_seconds/sample_count = null.
  - metrics — the measured quantities this card's history needs (kw, kvar, pf, voltage, current, iUnbalance, …), grounded in the card + the prompt metric. selection — the initial selected entity/section/bucket (or null). Leave a knob at its control default unless the prompt narrows it; null the OTHER family's knobs.
- fields[] — one resolved field per data slot, each:
    { slot, kind(raw|bucketed|time|derived|const|text|event), role(series|kpi|column|line|cell|tile|row|spoke),
      metric, column, label, unit, agg(avg|last|sum|count|derived), source(live|test-db|const|$ctx),
      value?(const only), base_columns?/sql_fragment?/nameplate_refs?(derived only), edge?(event only),
      sampling?(bucketed only), filters_table, has_data }
  `kind` is WHAT the field is; `source` is WHERE its data lives (set per R13, never by kind). `source` is the closed set live | test-db | $ctx | const — nothing else; 'const' only on a kind=const literal (or omitted there); a measured field is live/test-db/$ctx; 'derived' is a KIND, never a source.
  Per kind:
